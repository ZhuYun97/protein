Electricity-cost-aware multi-workflow scheduling in heterogeneous cloud

Abstract
Multi-workflows are commonly deployed on cloud platforms to achieve efficient computational power. Diverse task configuration requirements, the heterogeneous nature and dynamic electricity price of cloud servers impose significant challenges for economically scheduling multi-workflows. In this paper, we propose a Heuristic Electricity-cost-aware Multi-workflow Scheduling algorithm (HEMS) to search for an optimal scheduling plan which determines the optimal scheduling scheme for each task in each workflow, specifying the server to perform the task with determined resources in specific time. The objective is to minimize the total electricity cost of all servers while satisfying the deadline constraints of all workflows. The HEMS algorithm consists of five components: Workflow Scheduling Sequence Generation, Task Scheduling Sequence Initialization for each workflow, Optimal Scheduling Scheme Determination for each task, initial Task Scheduling Sequence Optimization, and Optimal Scheduling Plan Optimization. Experimental results demonstrate that HEMS consistently achieves the optimal scheduling plan with the lower total electricity cost (saving \(54.5–69.1\%\) on average) within slightly longer CPU time for various multi-workflows compared to existing three scheduling approaches.

1 Introduction
Complicated multi-workflow applications (e.g., bioinformatics science, astronomy and micro-service) are usually computationally intensive consisting of multiple independent heterogeneous workflows [1]. Each workflow is deadline-constrained specified by the owner and includes considerable diverse computing
tasks with data precedences. Efficient computing performance is achieved by scheduling the multi-workflow to a cloud datacenter with large-scale heterogeneous resources, such as Virtual Machines (VMs) and containers [2]. Different scheduling plans require different resource capacities which produce various energy consumption and Total Electricity Cost (TEC). The TEC constitutes a significant portion (more than \(30\%\) ) of the total operating cost of a cloud datacenter [3]. Therefore, it is desirable to attain an Optimal Scheduling Plan (OSP) which can produce a minimal TEC while meeting deadline constraints for all workflows to benefit both workflow owners and cloud providers.
The OSP determination in multi-workflow scheduling encounters challenges which mainly stem from the heterogeneous multiple workflows, the hybrid cloud resources and the dynamically fluctuated electricity price (EP for short). Specifically: (i) All tasks in the multi-workflow are scheduled sequentially following a Workflow Scheduling Sequence (WSS) and a Task Scheduling Sequence (TSS) for each workflow, which are determined based on specific rules. Various workflow deadlines, sophisticated task precedences and large-scale tasks in each workflow make it difficult to select the best rules to determine optimal sequences. (ii) Physical servers are virtualized as a heterogeneous resource pool consisting of VMs and containers configured with different CPU cores and memory capacities. Different types of tasks have diverse unknown resource requirements, making it tough to find the most appropriate server with the cheapest available time block for each task from the resource pool. (iii) The TEC is closely related to the EP which changes dynamically over time and is usually unknown in advance. It is essential but hard to approximately estimate the EP at a certain time.
However, most existing studies explore effective OSP determination techniques for single-workflows [2, 4–24] rather than multi-workflows. The few works on multi-workflow scheduling generally focus on VM-tasks executed on VMs with a constant EP [25–29]. They ignore another type of container-tasks deployed on lightweight containers and the dynamic nature of EP. In this paper, we consider an electricity-cost-aware heterogeneous multi-workflow scheduling problem with deadline constraints in a heterogeneous cloud datacenter with dynamic EP.
To solve the considered problem, a local-search based Heuristic Electricity-cost-aware Multi-workflow Scheduling algorithm (HEMS) is proposed to determine the OSP. The OSP can deploy each task in each workflow on the most viable server in the correct order at the suitable time. The main contributions are:
- A novel framework is proposed for scheduling multi-workflows, which incorporates three kinds of priority assignment rules: four rules for generating a WSS based on workflow characteristics, four rules for generating an initial TSS for each workflow considering task features and workflow structures, and three rules for selecting candidate servers for executing each task.- A dynamic-update based task sub-deadline distribution approach is embedded into HEMS to enhance the search efficiency for the OSP. HEMS applies the variable neighborhood descent algorithm and the container migration technique to reduce the probability of local optimization.
- Using the WorkflowSim simulation platform, we calibrate parameters and components through a comprehensive analysis of variance (ANOVA) involving multiple factors. This analysis is conducted on a large dataset of simulated and real multi-workflow instances. We perform numerous experiments and compare the results with existing algorithms to validate the superior performance of our proposed HEMS.
The rest of this paper is organized as follows. Section 2 reviews the related works. The multi-workflow scheduling architecture and the problem formulation are described in Sect. 3. HEMS is illustrated in Sect. 4 and Sect. 5 explains experimental results. Conclusions and future works are stated in Sect. 6.

2 Related work
The workflow scheduling in cloud environment is a typical NP-hard problem [30]. Global-exhausted-search based exact methods may find a global OSP but are computationally expensive and unacceptably time-consuming, especially for large-scale workflows. Existing researches usually apply heuristic algorithms [2, 4–10, 25, 26], meta-heuristic algorithms [11–18, 27], hybrid mechanisms [19–21], and Reinforcement Learning (RL) [28, 29]. Heuristic-based algorithms are usually limited to specific problems and may get stuck in local optimal solutions. Population-evolutionary-process based meta-heuristic algorithms provide greater generality and much better near-optimal solutions in different environments, such as Ant Colony Optimization (ACO) [11, 15], Particle Swarm Optimization (PSO) [12, 16], Glowworm Swarm Optimization (GSO) [13], and Genetic Algorithm (GA) [14, 27]. Hybrid mechanisms combine the advantages of both heuristic techniques and meta-heuristic algorithms to improve solution quality and search efficiency. For example, PA (Predict earliest finish time heuristic and ACO) and HA (Heterogeneous earliest finish time heuristic and ACO) were proposed in [20] to minimize makespan and cost. Multi-objective scheduling problems are also tackled using RL-based techniques such as coalition RL [28] and multi-agent RL [29].
The above works primarily focus on scheduling workflows with only VM-tasks which overlook the excellent benefits of containers in a cloud datacenter [2, 5–7, 11–14, 19–21, 25–29]. Containers are more flexible fine-grained virtual resources and offer higher efficiency for task execution. Substantial approaches have also been designed for scheduling containers for container-tasks [10, 16, 17, 22, 23]. However, tasks within a workflow usually have different natures which should be executed on different types of resources. For example, VMs with large storage capacity and strong stability are better for tasks with data consistency requirement. Containers are more suitable for tasks that support parallel processing. Therefore, it is necessary to design sophisticated mechanisms for hybrid scheduling of both VM-tasks and container-tasks as in most real-world scenarios. But only a minority of relevant studies have been conducted [10].
The constant EP is generally considered in most electricity-cost-aware workflow scheduling problems which actually cannot hold in many realistic scenarios [4, 5,
10–13, 19, 20, 25–29]. The objective of minimizing the TEC essentially degrades into minimizing the energy consumption. It is more practical to pay attention to the dynamics of EP while minimizing the TEC [2, 18, 21, 24]. For example, Zhang et al. adopted different scheduling algorithms under different change patterns of EP [24], i.e., a threshold-based mechanism for arbitrary patterns and a dynamic-programming-based mechanism for patterns following the Markov process.
Moreover, these scheduling approaches mainly target traditional single-workflows [2, 4–24]. They may suffer performance degradation when directly applied to heterogeneous multi-workflow scenarios which are more complex due to different deadline priorities and resource sharing. However, there are only a few multi-workflow scheduling approaches, and they only consider VM-tasks and constant EP [25–29]. Therefore, our approach HEMS focuses on scheduling the multi-workflow with hybrid tasks on a cloud datacenter with dynamic EP.

3 System framework and problem formulation
For the considered multi-workflow scheduling problem, this section presents the system framework and provides detailed mathematical models and problem formulation.

3.1 System framework
The complete framework of scheduling a heterogeneous multi-workflow to a heterogeneous cloud datacenter is depicted in Fig. 1. There are three layers: the User Layer, the Resource Layer and the Scheduling Layer. In the User Layer, users submit workflows with various characteristics which collectively form a heterogeneous multi-workflow. Each workflow is assigned a unique deadline based on user requirements. The number and dependencies of tasks within each workflow vary, resulting in diverse topology structures. Moreover, different workflows exhibit distinct task compositions, i.e., proportions of container-tasks and VM-tasks. In the Resource Layer, a virtual resource pool within the cloud datacenter hosts multiple computing nodes and one storage node. Computing nodes are heterogeneous containers and VMs with different CPU cores and memory sizes which are created by virtualizing physical servers. Only one data storage node is adopted to avoid redundant calculations and eliminate the data transferring time between any two tasks. The data transfer time between any one task and the data storage node is also neglected. The core Scheduling Layer bridges the gap between the two other layers including three modules: the Task Manager, the Scheduler, and the Monitor. The Task Manager determines the WSS and generates an initial TSS for each workflow based on specific priority assignment rules. The Scheduler obtains real-time updates on current EP and available computing resources from the Monitor to determine the Optimal Scheduling Scheme (OSS) for each task in the initial TSS. The OSS specifies the cheapest free time block on the available server for executing the task. Moreover, the Scheduler optimizes the initial TSS to avoid local optimization and generates the

3.2 Problem formulation
Assuming that the multi-workflow \(\mathbb{G}=\{G_{1},\cdots,G_{N}\}\) with \(N\) independent workflows is submitted in the User Layer. Each workflow \(G_{w}=\{V_{w},F_{w}\}(1\leq w\leq N)\) consists of \(n_{w}\) tasks with precedence constrains which is displayed by a directed acyclic graph (DAG) as shown in Fig. 1. \(V_{w}=\{\nu_{1}^{w},\cdots,\nu_{n_{w}}^{w}\}\) is the set of tasks, in which \(\nu_{1}^{w}\) is the start task and \(\nu_{n_{w}}^{w}\) is the end task. \(E_{w}=\{(\nu_{i}^{w},\nu_{j}^{w})|\nu_{i}^{w},\nu_{j}^{w}\in V_{w},i<j\}\) defines the partial order relation between two tasks, i.e., \(\nu_{j}^{w}\) can only be executed after its direct predecessor task \(\nu_{i}^{w}\) has been completed. Each task is modeled as \(\nu_{i}^{w}=(\mathcal{W}_{i}^{w},\beta_{i}^{w},\mathcal{P}_{i,w}^{u},\mathcal{R}_{i,w}^{c p u},\mathcal{R}_{i,w}^{m e m})(1\leq i\leq n_{w})\). \(\mathcal{W}_{i}^{w}\) denotes the workload size (i.e., instruction number). A binary variable \(\beta_{i}^{w}\) indicates the type, i.e., \(\beta_{w}^{i}\) takes 1 if and only if \(\nu_{i}^{w}\) is a VM-task and \(\beta_{w}^{i}=0\) otherwise. \(\mathcal{P}_{i,w}^{u p}\) represents the maximum parallelism that \(\nu_{i}^{w}\) can support. \(\mathcal{R}_{i,w}^{c p u}\) and \(\mathcal{R}_{i,w}^{m e m}\) are the CPU core number and memory size required for \(\nu_{i}^{w}\), respectively.
Let \(M\) heterogeneous physical servers in the Resource Layer be \(\mathbb{H}=\{H_{1},\cdots,H_{M}\}\). Each server is represented as \(H_{s}=(R_{s}^{c p u},R_{s}^{m e m},F_{s},P_{s}^{i d l e},P_{s}^{m a x})(1\leq s\leq M)\) consisting of five attributes: the total CPU core number, the total memory size, the instruction execution speed (i.e., main frequency), the power when \(H_{s}\) is idle and when \(H_{s}\)
is fully loaded. Once it is determined to schedule \(\nu_{i}^{w}\) to \(H_{s}\), the binary decision variable \(\alpha_{i,w}^{s}\) takes 1 and it is 0, otherwise. The actual execution time \(\mathcal{T}_{i,w}^{a e t}\) can be determined by \(\mathcal{W}_{i}^{w},F_{s}\), and the number of CPU cores \(\tilde{\mathcal{R}}_{i,w}^{c p u}\) that \(H_{s}\) actually assigns to \(\nu_{i}^{w}\). The VM startup time \(\tau\) is also considered if \(\nu_{i}^{w}\) is a VM-task.
The actual finish time \(\mathcal{T}_{i,w}^{a f t}\) is the sum of \(\mathcal{T}_{i,w}^{a e t}\) and the actual start time \(\mathcal{T}_{i,w}^{a s t}\).

4.1 Workflow scheduling sequence generation
Different WSSs generate different OSPs and TECs. It is necessary to design feasible priority assignment rules to sort \(\mathbb{G}\) to determine the proper WSS based on characteristics of workflows, such as deadline \(\mathcal{D}_{w}\), workload size \(\mathcal{W}_{w}\), and floating time \(\mathcal{T}_{w}^{float}\). For each workflow \(G_{w}\) in \(\mathbb{G}\), \(\mathcal{W}_{w}\) is the sum of the workload size of all its tasks, i.e., \(\mathcal{W}_{w}=\sum_{i=1}^{n_{w}}\mathcal{W}_{i}^{w}\). \(\mathcal{T}_{w}^{float}\) is the difference between \(\mathcal{D}_{w}\) and earliest finish time \(\mathcal{T}_{w}^{eft}\), i.e., \(\mathcal{T}_{w}^{float}=\mathcal{D}_{w}-\mathcal{T}_{w}^{eft}\). \(\mathcal{T}_{w}^{eft}\) is the maximal earliest finish time \(\mathcal{T}_{i,w}^{eft}\) among all tasks. For each task \(\nu_{i}^{w}\) in \(G_{w}\), \(\mathcal{T}_{i,w}^{eft}\) is the sum of actual execution time \(\mathcal{T}_{i,w}^{aet}\) and earliest start time \(\mathcal{T}_{i,w}^{est}\), i.e., \(\mathcal{T}_{i,w}^{eft}=\mathcal{T}_{i,w}^{aet}+\mathcal{T}_{i,w}^{est}\). However, \(\mathcal{T}_{i,w}^{aet}\) is unknown until \(\nu_{i}^{w}\) is completed. A feasible approach is to estimate \(\mathcal{T}_{i,w}^{aet}\) by assuming that \(\nu_{i}^{w}\) is always assigned to the server with the fastest single-core processing speed (i.e., \(F_{s}=F^{max}\)) and is always executed based on the maximum parallelism that \(\nu_{i}^{w}\) can support (i.e., \(\mathcal{R}_{i,w}^{cpu}=\mathcal{P}_{i,w}^{up}\)). Mathematically,
\(\mathcal{T}_{i,w}^{est}\) is obtained using the critical-path based dynamic programming method [32]:
\(\mathcal{T}_{i,w}^{e s t}\) is essentially the maximum earliest finish time \(\mathcal{T}_{i,w}^{e\bar{t}}\) among all predecessor tasks while neglecting the data transferring times between tasks.
Based on these features, four alternative priority assignment rules are proposed:
- Largest Workload First (LWF): The workflow with a larger \(\mathcal{W}_{w}\) has a higher priority.- Earliest Deadline First (EDF): The workflow with a smaller \(\mathcal{D}_{w}\) has a higher priority. For workflows with the same \(\mathcal{D}_{w}\), the one with a larger \(\mathcal{W}_{w}\) has a higher priority.- Smallest Floating time First (SFF): The workflow with a smaller \(\mathcal{T}_{w}^{f o a t}\) has a higher priority. For workflows with the same \(\mathcal{T}_{w}^{f o a t}\), the one with a larger \(\mathcal{W}_{w}\) has a higher priority.- Weighted Deadline and Floating time (WDF): Comprehensively considering EDF and SFF, the priority given by WDF is equal to the sum of priorities obtained from EDF and SFF.
An appropriate WSS is generated based on the descending order of priorities determined by the selected rule (discussed in Sect. 5.1). Each workflow is independently processed according to the WSS.

4.2 Task scheduling sequence initialization
Task precedences imply that a task requires the data produced by its direct predecessor tasks. The data waiting time is generated as long as any predecessor task has not been completed. Different TSSs in each workflow generate distinct total data waiting times of all tasks, leading to different server running schemes and TECs. It is essential to search for an optimal TSS. Exhaustively enumerating all possible TSSs may be time-consuming due to the large-scale of tasks. A more effective alternative manner is to determine an initial TSS and optimize it to generate a superior one. The initial TSS is established by sorting task priorities according to task characteristics, some of which are determined based on the sub-deadline \(d_{i}^{w}\).

4.2.1 Sub-deadline distribution
For each task \(\nu_{i}^{w},d_{i}^{w}\) is distributed based on the depth \(\ell_{i}^{w}\) and \(\mathcal{T}_{w}^{f o a t}\) [9]. \(\ell_{i}^{w}\) is recursively calculated from the start task \(\nu_{1}^{w}\) [32].
The depth \(\ell^{w}\) of the entire workflow is equal to the maximum depth of all tasks, i.e., \(\ell^{w}=\operatorname*{max}_{i\in\{1,\cdots,n_{w}\}}\{\ell_{i}^{w}\}\). The floating time \(\mathcal{T}_{i,w}^{f o a t}\) of each task is obtained by dividing
\(T_{w}^{f l o a t}\) in proportion to the ratio of \(\ell_{i}^{w}\) to \(\ell^{w}\) . \(d_{i}^{w}\) is the sum of \(T_{i,w}^{f l o a t}\) and the earliest finsh time Tef.
When the OSS of \(\nu_{i}^{w}\) is determined, the earliest start time \(\mathcal{T}_{i,w}^{e s t}\) is the actual start time \(\mathcal{T}_{i,w}^{a s t}\) . \(\tilde{\mathcal{T}}_{i,w}^{e t}\) is calculated by applying an assumption-based aggressive strategy (see (Eq. 9)) which may be smaller than the actual execution time \(\mathcal{T}_{i,w}^{a e t}\) . Consequently, \(\mathcal{T}_{i,w}^{e f t}\) may also be smaller than the actual finish time \(\mathcal{T}_{i,w}^{e f t}\) because \(\mathcal{T}_{i,w}^{e s t}+\tilde{\mathcal{T}}_{i,w}^{e t}=\mathcal{T}_{i,w}^{e f t}\) and \(\mathcal{T}_{i,w}^{a s t}+\mathcal{T}_{i,w}^{a e t}=\mathcal{T}_{i,w}^{e f t}\) . For any task \(\nu_{i^{\prime}}^{w}\) in the direct successor set \(s u c_{i}^{w}\) of \(\nu_{i}^{w},d_{i^{\prime}}^{w}\) may be over-tight since it is obtained essentially based on \(\mathcal{T}_{i,w}^{e f t}\) (deduced from Eqs. (10) and (12)) rather than \(\mathcal{T}_{i,w}^{e f t}\) . Therefore, a Dynamic Update based Sub-deadline Distribution algorithm (DUSD) is proposed to update \(d_{i^{\prime}}^{w}\) once \(\nu_{i}^{w}\) 's OSS is determined and \(\mathcal{T}_{i,w}^{e f t}>\mathcal{T}_{i^{\prime},w}^{e s t}\) . 4.2.2 Initial task sequence generation
Four alternative priority assignment rules are proposed based on task characteristics and precedence constraints:
- Maximum Parallelism First (MPF): Only the task characteristic is considered. The task with a higher maximum parallelism \(\mathcal{P}_{i,w}^{up}\) is given a higher priority.- Maximum Resource Requirement First (MRRF): Only the task characteristic is considered. Given that each task \(\nu_{i}^{w}\) requires both CPU cores and memory, an
index \(\mathcal{R}_{i}^{w}\) is introduced to express the resource requirement of \(\nu_{i}^{w}\), i.e., \(\mathcal{R}_{i}^{w}=\frac{\mathcal{R}_{i,w}^{c p u}}{R_{s}^{c p u}}+\frac{\mathcal{R}_{i,w}^{m e t}}{R_{s}^{m e t}}\). The task with a larger \(\mathcal{R}_{i}^{w}\) is given a higher priority.
- Minimum Execution Time First (METF): Only the task characteristic is considered. The unknown \(\mathcal{T}_{i,w}^{a e t}\) can also be approximated as the average execution time \(\bar{T}_{i^{\prime},w}^{e t}\) [33]. \(F^{m a x}\) in Eq. (9) is replaced with the average execution speed \(F^{a\nu g}\) of all servers to obtain \(\bar{T}_{i^{\prime},w}^{e t}\). The priority \(r a n k(\nu_{i}^{w})\) is recursively calculated based on \(\bar{T}_{i^{\prime},w}^{e t}\) in a similar manner as Eq. (10). The task with a smaller \(r a n k(\nu_{i}^{w})\) is given a higher priority.
- Shortest Slack Time First (SSTF): Both the task characteristic and task precedences are considered. The slack time \(\mathcal{T}_{i,w}^{s l a c k}\) indicates the longest time that \(\nu_{i}^{w}\) can be delayed before missing its sub-deadline. The task with a smaller \(\mathcal{T}_{i,w}^{s l a c k}\) is more urgently scheduled which is given a higher priority. \(\mathcal{T}_{i,w}^{s l a c k}\) is the difference between latest start time \(\mathcal{T}_{i,w}^{t s t}\) and earliest start time \(\mathcal{T}_{i,w}^{e s t}\). \(\mathcal{T}_{i,w}^{s t}\) is recursively calculated from the end task \(\nu_{n_{w}}^{w}\) [32].
An initial TSS is generated based on the descending order of priorities determined by the selected rule (discussed in Sect. 5.1). Each task is independently processed according to the TSS.

4.3 Optimal scheduling scheme determination
The OSS \(\Theta_{i,w}^{*}\) declares the server and the allocated number of CPU cores for executing \(\nu_{i}^{w}\) within a specific time block. To enhance the search efficiency for \(\Theta_{i,w}^{*}\), a two-phase strategy is adopted instead of the global search method. Specifically, (i) searching for available candidate servers \(\mathbb{H}_{i,w}^{C}(|\mathbb{H}_{i,w}^{C}|=S)\), (ii) searching for \(\Theta_{i,w}^{*}\) from \(\mathbb{H}_{i,w}^{C}\).

4.3.1 Available candidate server search
Three alternative server sorting rules are proposed based on server characteristics:
- Minimum Average Electricity Cost First (MAECF): Servers are sorted in ascending order based on the average electricity cost \(\bar{C}_{s}\) between \(\mathcal{T}_{i,w}^{e s t}\) and \(d_{i}^{w}\), i.e., \(\bar{C}_{s}=\int_{\mathcal{T}_{i,w}^{e s t}}^{d_{i}^{w}}P_{s,t}\times c(t)d t/(d_{i}^{w}-\mathcal{T}_{i,w}^{e s t})\).
- Maximum Total Computing Ability First (MTCAF): Servers are sorted in descending order based on the total computing ability \(A_{s}\) of all idle CPU cores between \(\mathcal{T}_{i,w}^{e s t}\) and \(d_{i}^{w}\), i.e., \(A_{s}=\int_{\mathcal{T}_{i,w}^{e s t}}^{d_{i}^{w}}F_{s}\times R_{s,t}^{idlecpu}d t\).
- Maximum Average Idle Resource First (MAIRF): Servers are sorted in descending order based on the average idle CPU and memory resource \(\bar{R}_{s}\) between \(\mathcal{T}_{i,w}^{est}\) and \(d_{i}^{w}\), i.e., \(\bar{R}_{s}=\int_{\mathcal{T}_{i,w}^{est}}^{d_{i}^{w}}(R_{s,t}^{idlecpu}+R_{s,t}^{idlemem})dt/(d_{i}^{w}-\mathcal{T}_{i,w}^{est})\).

The Available Candidate Server Search algorithm (ACSS) is proposed to sequentially check all servers in the sorted \(\mathbb{H}\) to determine \(\mathbb{H}_{i,w}^{C}\). 4.4 Task scheduling sequence optimization
The initial TSS (denoted as \(\pi_{w}\)) determined in Sect. 4.2.2 is not always the optimal sequence. In other words, the OSP \(\Psi_{\pi_{w}}=\{\Theta_{i,w}^{*}|1\leq i\leq n_{w},1\leq w\leq N\}\) obtained based on \(\pi_{w}\) may only be a local optimal solution, and the produced TEC \(\mathcal{C}_{\pi_{w}}=\sum\mathcal{C}_{i,w}^{*}\) may not be minimal. Therefore, TSSO is proposed based on the Variable Neighborhood Descent (VND) algorithm to explore a better TSS with a lower TEC. Algorithm 5 is the formal description of TSSO. Firstly, TSSO generates \(K\) new neighborhood sequences by performing the task-pair-swap action, i.e., randomly selecting two nonrepetitive tasks and swapping their positions in \(\pi_{w}\). The selection of task pairs must meet two conditions: (i) To ensure the task precedence constraint as expressed in Eq. (6), only task-pairs with the same level (i.e., depth in Eq. (11)) can be chosen. (ii) The number of task-pairs should balance probability and search time to find a better OSP faster. TSSO adopts a level-space-reduction based mechanism which only selects task-pairs from the first \(K-k+1\) levels for the \(k\)-th neighborhood sequence. Smaller level space implies less number of alternative task-pairs. Next, for each generated neighborhood sequence \(\pi_{w}^{\prime}\), two algorithms (OSSS and ACSS) are recursively called to calculate the TEC \(\mathcal{C}_{\pi_{w}^{\prime}}\). The optimized TSS \(\pi_{w}^{*}\) corresponds to the one that produces the minimal TEC among all \(\pi_{w}^{\prime}\) and \(\pi_{w}\). The time complexity of TSSO is \(O(Kn_{w})\) to obtain \(\pi_{w}^{*}\). \(K=\gamma\ell^{w}\) is determined based on a linear relationship with the workflow depth \(\ell^{w}\). The optimal value for \(\gamma\) is discussed in Sect. 5.1.

5 Performance evaluation
Our proposed HEMS is compared with three existing scheduling algorithms [7, 33, 34]. All four algorithms are implemented on the simulated WorkflowSim platform [35], which includes three layers: the User Layer, the Scheduler Layer, and the Resource Layer (see Fig. 1). The User Layer uses a Workflow Generator tool [36] to generate various multi-workflows to prove the effectiveness and generalization of HEMS. They are four types: (i) CyberShake-type uniform multi-workflows consisting of CyberShake workflows, (ii) Epigenomics-type uniform multi-workflows consisting of Epigenomics workflows, (iii) LIGO-type uniform multi-workflows consisting of LIGO workflows, (iv) hybrid multi-workflows consisting of all three types of workflows. Both data-intensive CyberShake and CPU-intensive LIGO are highly parallel, and more sink tasks exist in LIGO. The data-pipeline-like Epigenomics is a combination of parallel and sequential tasks. Their topology structure representations are detailed in [19, 34]. Each multi-workflow contains \(N \in \{20, 40, 60, 80, 100\}\) independent workflow instances. For each instance \(G_{w}\), the number of tasks is \(n_{w} \in \{50, 100, 200, 500\}\), the proportion of container tasks is \(\mu \in \{0, 0.2, 0.4, 0.6, 0.8, 1\}\), and the deadline is \(\mathcal{D}_{w} = \mathcal{T}_{w}^{eff} + \mathcal{T}_{w}^{efi} \times \xi\),
where \(\xi \in \{0.2, 0.4, 0.6, 0.8, 1\}\) is a deadline factor. 10 instances are generated for each combination of number, proportion, and factor. In total, there are \(5 \times 4 \times 6 \times 5 \times 10 = 6000\) instances for performance comparisons.
Eight types of high-performance servers are available in the Resource Layer which have different configurations as listed in Table 2. The EP of servers fluctuates dynamically similar to that in the actual scenario of Toronto, Ontario, Canada [37] as depicted in Fig. 2. The number of servers of each type is random but fixed. Each server is virtualized as containers and VMs. The number of CPU cores allocated for containers is \(\{0.25, 0.5, 1, 2, 4\}\) while for VMs is \(\{1, 2, 4, 8\}\), which is consistent with Amazon EC2's instance specification. The startup time of VMs is set to \(\tau = 30\) seconds.

5.1 Component calibration
In the Scheduler Layer, HEMS determines the OSP for the multi-workflow. There are five components requiring calibration: (i) four priority assignment rules for generating the WSS in the WSSG component, including LWF, EDF, SFF and WDF; (ii) four priority assignment rules for generating the initial TSS for each workflow in the TSSI component, including MPF, MRRF, METF and SSTF; (iii) three sorting rules for selecting candidate servers in the OSSD component, including MAECF, MTCAF and MAIRF; (iv) the parameter \(\gamma \in \{0, 0.25, 0.5, 0.75, 1.0\}\) for determining the number of neighborhood sequences in the TSSO component; (v) the parameter \(\varphi \in \{5\%, 10\%, 15\%, 20\%, 25\%\}\) for determining inefficient servers in the OSPO component. Different combinations of these three rules and two parameters result in different OSPs, whose best values are calibrated through the multi-factor analysis of variance (ANOVA) technique [38]. 5.2 Experimental results
Three factors can affect the performance of scheduling algorithms which include task urgencies (i.e., sub-deadlines), task precedences, and server EP. Our proposed HEMS comprehensively considers all factors. The compared three algorithms consider different factors and adopt different mechanisms to generate the optimal TSS for each workflow \(G_{w}\) and determine the task-server mapping:
- HEFT (Heterogeneous Earliest Finish Time) [33] generates the TSS in descending order of \(rank(\nu_{i}^{w})\), \(rank(\nu_{i}^{w})\) is the sum of the execution time of \(\nu_{i}^{w}\) and the maximal execution time among all its immediate successor tasks. \(\nu_{i}^{w}\) is assigned to the server with the faster execution speed. Only local task precedences are considered.- TUMS-WTM (Time Utilization Maximization Scheduling-Working Time Minimization) [7] generates the TSS in descending order of task priorities. A higher priority is given based on less slack time, more dependent tasks and lower topological level. The task-server mapping is achieved based on the start execution time. Only task urgencies and task precedences are considered.- DQWS (Divide-and-conquer Workflow Scheduling) [34] first schedules tasks in the critical path to the same cheapest server which can meet \(\mathcal{D}_{w}\). These tasks are removed and \(G_{w}\) is split into multiple sub-workflows. This process is repeated until only the chain sub-workflows remain. All three factors are considered.

However, for larger instances, the divide-and-conquer based TSS generation mechanism of DQWS generates more hung-up tasks which causes TUMS-WTM to outperform DQWS. Figure 6b indicates that all four algorithms are much more suitable for instances with more container-tasks. HEMS obtains the lowest RPD and reaches a stable value at \(\mu=0.8\). Figure 6c shows that the deadline factor \(\xi\) does not affect the performance of HEFT while DQWS outperforms TUMS-WTM. These results are consistent with those for CyberShake-type multi-workflows since they are both highly parallel. Larger \(\xi\) produces smaller RPDs for the other three algorithms.

5.2.2 Hybrid multi-workflows
Fig. 7 depicts the comparison results of all four algorithms for the hybrid multi-workflows with different characteristics (i.e., \(n_{w}\), \(\mu\) and \(\xi\)). Similar to the results for the three types of uniform multi-workflows, HEMS outperforms the other three algorithms while HEFT is always worst-performing for hybrid multi-workflows. DQWS assigns all critical tasks to the same server to eliminate inter-task communication time. This local-scheduling based mechanism may result in more queuing time and violations of workflow deadlines due to the limited capacity of servers. Therefore, DQWS consistently performs worse than TUMS-WTM regardless of variations in \(n_{w}\) and \(\mu\) as shown in Figs. 7a and b. As \(\xi\) increases, DQWS is provided with looser deadlines. The EP-based server assignment mechanism can take greater advantage to find OSPs with smaller TECs. A lower RPD is always achieved by DQWS than TUMS-WTM as shown in Fig. 7c. The deadline factor \(\xi\) exerts little influence on HEFT's performance due to the execution-speed based server assignment mechanism. All four algorithms basically perform better for hybrid multi-workflows with fewer tasks, higher container-task proportions, and looser deadlines. The main reasons are: (i) More tasks imply more complex workflow structures, making it harder to find an optimal TSS. (ii) More container-tasks result in higher resource utilization. (iii) Looser deadlines provide a wider search space in finding the OSP with minimal TEC.

5.2.3 Comparisons of average RPDs and CPU Times
Table 3 summarizes the average RPDs and CPU times of all four algorithms for four types of multi-workflows. Since HEFT only considers a single influence factor (i.e., local task precedences), it always obtains the highest average RPDs but the fastest computing speeds. HEMS considers all three factors (i.e., task urgencies, task precedences and server EP) which always achieves the lowest average RPDs with slightly longer CPU times. Both DQWS and TUMS-WTM outperform HEFT but perform worse than HEMS. They are suitable for different types of multi-workflows. DQWS exhibits better performance and obtains lower RPDs for Epigenomics-type uniform multi-workflows with many sequential tasks and hybrid multi-workflows.
TUMS-WTM achieves lower RPDs for CyberShake-type and LIGO-type uniform multi-workflows with many parallel tasks. Overall, HEMS achieves RPD reductions of about \(69.1\%\), \(55.0\%\), and \(54.5\%\) compared to HEFT, DQWS, and TUMS-WTM, respectively.

6 Conclusions and future work
In this paper, we consider a more realistic multi-workflow scheduling problem with hybrid tasks and dynamic electricity price of servers. A local-search based static heuristic optimization algorithm HEMS is proposed and compared with three adapted state-of-the-art algorithms. The experimental and statistical analyses reveal that HEMS is much more suitable for multi-workflows with fewer tasks, more container-tasks, and looser deadlines. Moreover, HEMS consistently achieves the optimal scheduling plan with a lower total electricity cost within a reasonable CPU time for both uniform and hybrid multi-workflows. The future research focus on multi-workflow scheduling that considers task dependencies across workflows and time-varying communication times between tasks.
